# skill-eng-manager-1on1

A profession-shaped NeuroDock skill for the three things engineering managers, tech leads, and staff+ ICs with mentees actually do in their week: prep recurring 1:1s, decompose sprint reviews into atomic items, and walk blameless postmortems. Three trigger phrases, three flows, no overlap with the general-purpose planning skills.

## Who this is for

- **Engineering managers** running recurring 1:1s with reports and trying to keep track of open threads across five or ten people.
- **Tech leads** who run sprint reviews/retros and need to turn an hour of unstructured discussion into concrete next steps.
- **Staff+ ICs with formal or informal mentees** — same shape as a manager's 1:1, just without the reporting line.
- **Anyone running blameless postmortems** on incidents and looking for a tool that helps them stay blameless when the room wants someone to blame.

It is **not** for:
- Performance reviews, PIPs, calibration, or any HR cycle. This skill does not produce performance summaries and is deliberately blind to anything that would feed one.
- Sprint *planning*. This skill decomposes a *review* — what was done — into action items. The roadmap is a different tool.
- Therapy or coaching. The skill helps you structure a conversation; it does not coach you on how to have it.

## What it does

Three activation paths. The trigger phrase determines the flow.

| Trigger | Flow | What it produces |
|---|---|---|
| `prep my 1:1 with <name>` | 1:1 prep | A one-page brief built from prior context recorded in `mcp-cognitive-graph` — last conversation timestamp, open topics, recent related projects, one suggested opener. |
| `decompose this sprint review` | Sprint-review decomposition | Atomic action items (5–90 minutes each, with acceptance criteria) grouped by owner, generated by `neurodock-task-fractionator`. |
| `blameless postmortem on <incident>` | Postmortem | A structured 5 Whys + system-factors skeleton that never names a person as cause. |

See `SKILL.md` for the full operating instructions and triggers.

## What it explicitly does NOT do

- It does not score, rank, or rate anyone.
- It does not write a performance summary, ever.
- It does not propose PIPs, escalations, or HR actions.
- It does not name a person as the cause of an incident, even when the user asks it to.
- It does not use middle-management vocabulary (no "synergy", "rockstar", "10x", "ninja", "stretch goal", "growth mindset", "should have known").
- It does not call `record_fact` without explicit user opt-in. The cognitive graph is the user's notebook.

## Why these three flows together

These are the recurring meetings an engineering manager runs that benefit from structure but cannot be templated. They share a common shape: take messy human input (memory of past conversations, unstructured notes, an emotional incident debrief), produce a structured artifact that the user can act on, then stop. The skill is profession-shaped — it would not make sense to factor each flow into its own plugin, because the user who wants one almost always wants the others.

## Dependencies

| MCP server | Required? | Used for |
|---|---|---|
| `neurodock-task-fractionator` | **Hard** | The `decompose` tool in Flow B. Without it, sprint-review decomposition cannot run. |
| `mcp-cognitive-graph` | Optional | `recall_entity` for Flow A 1:1 prep, `record_fact` for opt-in postmortem capture in Flow C. Without it, Flow A falls back to a "what's on your mind?" prompt. |
| `mcp-chronometric` | Optional | `mark_session_start` / `mark_session_end` for users who want to time-box the prep. The skill works without it. |

The hard requirement is declared in `plugin.yaml` under `requires.mcp_servers`. The two optional integrations are declared in `SKILL.md` frontmatter with `optional: true` and are gated behind availability checks in the operating instructions.

## How to install

NeuroDock discovers plugins from two filesystem roots. To install for local testing:

**Per-user (recommended for trying it out):**

Linux:
```bash
mkdir -p ~/.local/share/neurodock/plugins/
cp -r plugins/skill-eng-manager-1on1 ~/.local/share/neurodock/plugins/
```

macOS:
```bash
mkdir -p "$HOME/Library/Application Support/neurodock/plugins/"
cp -r plugins/skill-eng-manager-1on1 "$HOME/Library/Application Support/neurodock/plugins/"
```

Windows PowerShell:
```powershell
$dest = "$env:APPDATA\neurodock\plugins\skill-eng-manager-1on1"
New-Item -ItemType Directory -Force -Path (Split-Path $dest) | Out-Null
Copy-Item -Recurse plugins\skill-eng-manager-1on1 $dest
```

**In-repo (already done if you're reading this from the cloned repo):**

The substrate also scans `<repo>/plugins/*/plugin.yaml`, so cloning this repo is enough — no copy needed.

Restart your LLM client (Claude Desktop, Claude Code). The substrate scans both roots at init.

To verify discovery:
```bash
neurodock plugin list
```

The plugin should appear with `trust: community` and the three triggers listed above.

## How to use

Once installed, just say one of the trigger phrases in your normal LLM client. The skill is profile-agnostic (`neurotypes: []`) so it auto-activates for any user whose profile allows community plugins.

Examples:

> Prep my 1:1 with Sarah.

> Decompose this sprint review:
>
> [paste notes]

> Blameless postmortem on the cache outage from last Tuesday.

The skill replies with a structured brief, grouped action items, or a postmortem skeleton (respectively). It does not propose follow-up questions, ask "anything else?", or otherwise extend the conversation. When the artifact is ready, the skill stops.

## When NOT to use it

- **You're prepping a performance review.** This skill cannot help — by design. It does not produce performance summaries and will refuse to score, rank, or rate any person.
- **You want to name who's responsible for an incident.** The postmortem flow will not do this. If your organisation needs that conversation separately, route it separately.
- **You're planning the next sprint.** Different flow. `decompose this sprint review` operates on what was done; it does not generate a roadmap.
- **You're trying to "manage up" or write a status email.** Use a general-purpose planning or writing skill. This one is for the work, not for the optics.

## Pairing with other skills

- Pair with `adhd-daily-planner` (in-tree) if you want a morning brief that surfaces "you have a 1:1 with Sarah today" so you can chain into `prep my 1:1 with Sarah`.
- Pair with `audhd-context-recovery` (in-tree) when returning from a long absence — `where was I` first to reconstruct the project state, then `prep my 1:1 with <name>` for the upcoming conversation.
- See `examples/founders-morning-brief/` for a worked walkthrough of the morning ritual; founders with direct reports will want to chain that with this skill.

## Tests

Three replayable conversation tests in `tests/` cover the three flows end-to-end. Same format as `plugins/example-skill-pomodoro/tests/`. Run them with:

```bash
neurodock skill test plugins/skill-eng-manager-1on1
```

(Available once the reference replay harness ships with `@neurodock/skill-sdk`.)
