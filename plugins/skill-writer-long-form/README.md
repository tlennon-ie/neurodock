# skill-writer-long-form

A profession-shaped NeuroDock skill for the three things a writer working on something long actually does inside a working session: plan the next N hours as scene/section-shaped atomic tasks, recall what the graph remembers about a character or source or concept, and check honestly whether they are re-editing a chapter that should already be closed. Three trigger phrases, three flows, no overlap with the general-purpose planning skills.

The skill respects the slowness of real writing. It does not measure success in words per hour. When the rumination check fires, it does not offer to "take another look".

## Who this is for

- **Novelists, short-fiction writers, screenwriters** working a recurring writing block and trying to leave the desk with something concrete drafted or revised — not a stack of partial scenes.
- **Essayists and long-form journalists** building a 3,000–10,000-word piece across multiple sessions and needing to keep track of the through-line.
- **Technical-book and reference-book authors** writing chapters across weeks or months with a real character cast (the running examples, the customer personas, the named systems) the graph has to remember.
- **Dissertation writers** treating the dissertation as a long-form work — chapters that interlink, sources that recur across chapters, an internal contradiction in a character (or in a model, or in a citation) that surfaces in chapter 4 because of something promised in chapter 1.
- **Anyone who has reopened the same chapter for the fifth time this week** and wants an honest answer.

It is **not** for:

- **Productivity coaching.** This skill does not help anyone write faster. If anything it slows the user down by refusing to set word-count goals.
- **Content marketing or SEO writing.** Different shape of work, different success metric. The "10 hours of writing this week" model the skill assumes does not apply.
- **Manuscript critique.** It does not read your prose, comment on voice, or suggest structural revisions.
- **Therapy or self-help.** The rumination check is a heuristic delivered via mcp-guardrail; it is not a clinical instrument and the skill makes no claims about the user's neurotype.

## What it does

Three activation paths. The trigger phrase determines the flow.

| Trigger                                        | Flow                  | What it produces                                                                                                                                                                           |
| ---------------------------------------------- | --------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `plan today's writing block, <N> hours`        | Writing-block planner | A list of scene/section-shaped atomic tasks (5–90 minutes each, with acceptance criteria), generated by `neurodock-task-fractionator` from the time budget. **No word counts.**            |
| `who is <character/source/concept>`            | Recall                | What the graph remembers about the entity: prior notes verbatim, connected projects/chapters, and **internal contradictions you have accidentally introduced** — named, not smoothed over. |
| `I keep editing chapter <N>, am I ruminating?` | Rumination check      | A guardrail call to `check_rumination`. If detected, a verbatim closing line that refuses to re-validate the chapter. If not detected, a one-line "carry on".                              |

See `SKILL.md` for the full operating instructions and triggers.

## Framing — time and tasks, not words and pages

The whole reason this skill exists separately from a general planner is that long-form writing is not a throughput problem. Drafting a scene takes the time it takes. Revising it takes the time it takes. The visible artifact of an hour of real writing might be 1,200 words or 80, and the skill cannot tell which session was more valuable from the count alone — so it does not try.

What the skill helps with:

- **Shape, not volume.** The writing-block planner emits tasks like `draft scene 3: kitchen confrontation` or `revise the opening paragraph of chapter 2 for voice` — concrete, finite pieces of writing or discrete revision passes. Estimates are in minutes, not in words.
- **Continuity, not summary.** The recall flow surfaces what the graph remembers about a character or source as the user has recorded it, verbatim, with timestamps. It does not paraphrase, does not generate a "character profile", does not summarise an arc.
- **Honesty, not encouragement.** The rumination check defers detection to mcp-guardrail and, when it fires, returns a hard closing line. It does not call editing a virtue, does not praise care for the craft, does not offer to take another look.

## What this skill is NOT

- **Not a word-count tracker.** Word counts are a content-mill metric — they reward typing volume regardless of whether the writing is any good. Real writers in actual flow do not measure themselves in words/hour, and this skill refuses to either. The plan is in tasks and minutes. The closing line of every Flow A output is `These tasks are shaped to the time you have. Word counts are not in the plan, on purpose.`
- **Not a productivity hack.** It does not help anyone write faster, more "efficiently", or "more prolifically". If those framings are the goal, use a different tool.
- **Not a writing coach.** It does not comment on prose, voice, structure, arc, pacing, or any craft dimension. It supports the work; it does not critique the work.
- **Not a summarisation tool.** It does not generate plot summaries, character profiles, or chapter abstracts. The recall flow surfaces literal facts the user recorded; it does not synthesise.
- **Not a manuscript critique tool.** Use beta readers, editors, or workshop. Not this.
- **Not a "your daily writing dashboard" tool.** It does not show streaks, badges, "you wrote 5 days in a row", or any other gamification. The work is not a game.
- **Not a citation manager.** If you are writing a non-fiction book with sources, use Zotero for the bibliography. This skill records sources as graph entities so it can recall "who is Pollan", not so it can emit BibTeX.
- **Not a clinical tool.** The rumination check is a heuristic delivered via mcp-guardrail. It is not a diagnosis. It is generally useful to any writer who has been re-opening the same chapter all week.

## Why these three flows together

These are the three things any sustained long-form writing project requires inside a working session: a shape for the next few hours, a way to recover what was previously written down about a character or source, and an honest reading of whether the writer is making progress or churning. They share a common pattern (messy intent → structured artifact → stop), they reinforce each other across sessions (Flow A's tasks reference characters Flow B can recall; Flow C closes the door on the chapters Flow A no longer needs to plan around), and a writer who needs one almost always needs the others. Splitting them across three plugins would force the user to install three things to get one workflow.

## Dependencies

| MCP server                    | Required? | Used for                                                                                                                                                                                                        |
| ----------------------------- | --------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `neurodock-task-fractionator` | **Hard**  | The `decompose` tool in Flow A. Without it, the writing-block planner cannot break the time budget into scene-shaped tasks.                                                                                     |
| `neurodock-cognitive-graph`   | **Hard**  | `recall_entity` in Flow B (who is X). Without it, the recall flow cannot run — there is no useful fallback to "I don't know who X is".                                                                          |
| `neurodock-guardrail`         | Optional  | `check_rumination` in Flow C. Without it, Flow C refuses to fabricate a check and returns a structured one-line message. Flows A and B still work. The rumination check is the load-bearing ND-supportive flow. |

The two hard requirements are declared in `plugin.yaml` under `requires.mcp_servers`. The optional guardrail dependency is declared in `SKILL.md` frontmatter with `optional: true` and is gated behind an availability check in step 2 of Flow C.

## How to install

NeuroDock discovers plugins from two filesystem roots. To install for local testing:

**Per-user (recommended for trying it out):**

Linux:

```bash
mkdir -p ~/.local/share/neurodock/plugins/
cp -r plugins/skill-writer-long-form ~/.local/share/neurodock/plugins/
```

macOS:

```bash
mkdir -p "$HOME/Library/Application Support/neurodock/plugins/"
cp -r plugins/skill-writer-long-form "$HOME/Library/Application Support/neurodock/plugins/"
```

Windows PowerShell:

```powershell
$dest = "$env:APPDATA\neurodock\plugins\skill-writer-long-form"
New-Item -ItemType Directory -Force -Path (Split-Path $dest) | Out-Null
Copy-Item -Recurse plugins\skill-writer-long-form $dest
```

**In-repo (already done if you're reading this from the cloned repo):**

The substrate also scans `<repo>/plugins/*/plugin.yaml`, so cloning this repo is enough — no copy needed.

Restart your LLM client (Claude Desktop, Claude Code). The substrate scans both roots at init.

To verify discovery:

```bash
neurodock plugin list
```

The plugin should appear with `trust: community` and the three triggers listed above.

## How to use

Once installed, just say one of the trigger phrases in your normal LLM client. The skill is profile-agnostic (`neurotypes: []`) so it auto-activates for any user whose profile allows community plugins.

Examples:

> Plan today's writing block, 3 hours.

> Who is Marlene?

> I keep editing chapter 4, am I ruminating?

The skill replies with a task list, a recall block, or a rumination determination (respectively). It does not propose follow-up questions, ask "anything else?", or otherwise extend the conversation. When the artifact is ready, the skill stops.

## When NOT to use it

- **You want to hit a daily word-count target.** This skill refuses to set or track word counts. By design.
- **You want feedback on your prose.** Different tool. Try a beta reader or a workshop.
- **You want a chapter summary or character profile generated.** This skill records and recalls literal notes; it does not synthesise.
- **You want to be told that re-editing chapter 4 for the fifth time this week is fine.** It is not the skill's job to validate that. The rumination check exists exactly because validating that hurts more writers than it helps.

## Decisions flagged for human review

Three design choices in this skill are deliberate but worth a maintainer's second look before the plugin moves out of `0.1.0`:

1. **Internal-contradiction detection in Flow B is a deliberately simple keyword-pair rule.** The skill walks the entity's facts and flags pairs whose literals contain a matching descriptor keyword (`height`, `age`, `tall`, `hair`, `from`, `born in`, etc.) and disagree on the numeric or short-form part. This is intentionally crude in v0.1.0 — the alternative (full natural-language disagreement detection) is out of scope for a stateless skill and would require a separate clinical-grade analyser. **Maintainer review**: confirm the keyword list is large enough to be useful but small enough to avoid false positives; revisit when the cognitive graph gains a `contradicts` predicate.

2. **The verbatim closing line in Flow C is hard-coded and load-bearing.** `Chapter <N> is in the can. Open a new question if there is a NEW concern, or move to Chapter <N+1>.` is intentionally not configurable per-user. The reason: a writer who has rewired this line to soften it has rewired away the redirect that the rumination check exists to deliver. **Maintainer review**: confirm the line is right, or propose a small set of equally-firm variants — but do not make it user-tunable. The test asserts the verbatim line.

3. **`neurodock-guardrail` is a soft requirement, not hard.** Flow C is the load-bearing ND-supportive flow, so an argument can be made that the skill should refuse to install at all when the guardrail server is absent. The counter-argument (which won): Flows A and B remain genuinely useful in standalone mode, and forcing the user to install a server they cannot run for any other reason just to plan a writing block is bad UX. The skill therefore installs, exposes all three triggers, and Flow C returns a structured "guardrail not available" line when the server is absent rather than silently misbehaving. **Maintainer review**: confirm this trade-off, or escalate Flow C's guardrail dependency to hard in a future minor bump.

## Pairing with other skills

- Pair with `adhd-daily-planner` (in-tree) if you want a morning brief that surfaces today's writing block alongside other commitments, then chain into `plan today's writing block, <N> hours` for the block itself.
- Pair with `audhd-context-recovery` (in-tree) when returning to a writing project after a break — `where was I` first to reconstruct the project state, then `who is <character>` to refresh the cast before opening the manuscript.
- Pair with `ocd-decision-finalizer` (in-tree) if the rumination in Flow C turns out to be about a structural decision (e.g. POV, timeline, framing device) rather than about chapter prose. The decision-finalizer handles closed decisions; this skill handles closed chapters. Different surfaces, complementary posture.
